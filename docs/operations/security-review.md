# Security review (M5.1)

A walkthrough of the threat model in [MASTER_PLAN.md §8](../MASTER_PLAN.md#8-security-architecture),
mapping each control to where it is implemented and the automated check that keeps it
honest. The threat is that an in-home camera feed of a child is a high-value target;
the posture is **locked down by default, convenience opt-in**.

Legend: **[A]** = enforced by an automated test/CI gate (the regression harness);
**[M]** = manual / procedural.

## §8 controls

### 1. No internet exposure by default

Only Caddy publishes ports; every other service is on an internal Docker network. Remote
access is via WireGuard/Tailscale, never port-forwarding (the install docs warn against
exposing the stack).

- **[A]** `deploy/tests/test_stack.py::test_only_caddy_publishes_ports` — asserts no
  non-edge container publishes a host port.
- **[A]** `test_all_containers_hardened` — `no-new-privileges`, `cap_drop: ALL`,
  read-only rootfs, non-root users on every service.

### 2. TLS everywhere, including LAN

Caddy terminates HTTPS with a locally provisioned CA (generated by `install.sh`). MQTT is
TLS-only with per-device credentials. WebRTC media is DTLS-SRTP by protocol.

- **[A]** `test_http_redirects_to_https`, `test_tls_chains_to_local_ca_with_headers`
  (HSTS + security headers, chain to the local CA).
- **[A]** MQTT TLS + no-plaintext: `deploy/tests/sensors/test_sensors.py` (M3.1) — a
  plaintext or wrong-credential connect is refused.

### 3. Authentication

Argon2 password hashing + optional TOTP 2FA; short-lived HS256 JWT access tokens with
rotating, theft-detecting refresh-token families; per-device MQTT credentials with
topic-scoped ACLs; scoped API tokens for integrations.

- **[A]** `server/tests/test_auth_api.py` — login, refresh rotation, **refresh-reuse
  revokes the family**, logout revocation, TOTP, brute-force lockout, API-token scopes.
- **[A]** `server/tests/test_auth_fuzzing.py` (M5.1) — the **auth-fuzzing corpus**: token
  tampering (flipped signature, payload swap under the old signature, attacker-secret
  signature, wrong token type), downgrade (`alg:none` forgery, HS512 algorithm
  confusion), replay (expired access token, replayed rotated refresh token), and an
  escalation probe proving the token's `role` claim is **not** authoritative
  (authorization reads the database role).
- **[A]** `deploy/tests/sensors/test_sensors.py::test_device_acl_isolation` — a
  compromised node can publish only its own `eeper/dev/{id}/` subtree.

### 4. Authorization roles (admin / viewer "grandparent mode")

A viewer is scoped to Live + Tonight; settings, history export, and device management are
admin-only (both API and UI).

- **[A]** `server/tests/test_auth_api.py::test_viewer_denied_on_admin_endpoints`.
- **[A]** `web/e2e/roles.spec.ts` (`e2e-roles`, M4.3) — a viewer sees only Live + Tonight,
  is redirected from every management route, and is denied CSV export (403).

### 5. Container hardening & supply chain

Slim images, non-root, read-only rootfs, no privileged containers, only Caddy exposes
ports, pinned image digests, image scanning + dependency/secret scanning in CI.

- **[A]** `test_all_containers_hardened` (see §1).
- **[A]** Trivy image scan — `images.yml` fails the build on a CRITICAL CVE (both arches).
- **[A]** `dependency-audit` (M5.1) — `pip-audit` over the shipped runtime closure +
  `npm audit --omit=dev`.
- **[A]** `secrets-scan` (M5.1) — gitleaks over full history; `.gitleaks.toml` allowlists
  only dummy test fixtures + `*.example` templates.

### 6. Secrets

Generated at first run into `deploy/.env` (git-ignored); no default passwords anywhere;
the first-boot wizard forces admin credential creation.

- **[A]** `test_no_default_credentials_in_env`, `test_database_has_no_default_user_rows`,
  `test_first_boot_required_on_fresh_install`, `test_protected_endpoint_401_before_wizard`.
- **[A]** `secrets-scan` (M5.1) — no real credential is ever committed.

### 7. Privacy posture

All inference is local; no telemetry; no cloud dependency for any v1 feature. Optional,
gated inputs (pulse-ox) are insights-only and inert until explicitly enabled.

- **[A]** No outbound calls in the runtime path (the stack runs air-gapped in CI — every
  integration suite brings up the stack with no internet dependency).
- **[A]** Pulse-ox gating: `server/tests/test_pulseox_api.py` (M4.2) — inert without the
  profile flag **and** an admin acknowledgment.

## Auth-fuzzing corpus (the M5.1 [A] criterion)

`server/tests/test_auth_fuzzing.py` hits the live HTTP boundary with the cookies an
attacker would send. Every crafted token fails closed (401), and the escalation probe is
forbidden (403) because authorization never trusts a token's self-declared role. See the
list under §3.

## Residual risk & the manual pen test — [M]

The automated harness covers the _known_ attack classes above. Adversarial creativity is
not automatable, so M5.1 also requires a **[M] penetration test (or structured community
security review) of a default install** that finds no unresolved critical/high findings.

Status: **pending.** When performed, record the reviewer, date, scope, and the disposition
of every finding here. Any fix lands with a regression test so the harness grows to cover
what the pen test found.
