# M0.2 [MANUAL] — Trust the local CA on iOS and Android

**Criterion:** Trust the local CA on one iOS and one Android device following
this doc; the PWA then loads over HTTPS without certificate warnings. (Physical
devices; the usability of this doc is part of the check.)

**Record the result** in [PROGRESS.md](../../PROGRESS.md) next to the criterion,
e.g. `✔ 2026-08-14 JD`.

## Prerequisites

- The stack is running (`deploy/install.sh` finished) and reachable from the
  phone over the LAN at `https://<EEPER_DOMAIN>/`.
- The CA file `deploy/eeper-local-ca.crt` has been copied to the phone (AirDrop,
  a USB transfer, or serve it once over the LAN and download it).

## iOS / iPadOS

1. Open the `eeper-local-ca.crt` file on the device → tap **Allow** to download
   the profile.
2. **Settings → General → VPN & Device Management** → under _Downloaded Profile_
   tap the eeper CA → **Install** (enter passcode) → **Install** again.
3. **Settings → General → About → Certificate Trust Settings** → toggle **ON**
   for the eeper root certificate.
4. Open Safari and browse to `https://<EEPER_DOMAIN>/`.

**Pass:** the page loads with no certificate warning and a valid-lock indicator.

## Android

_(Steps vary slightly by OEM/version; this targets stock Android 13+.)_

1. Copy `eeper-local-ca.crt` to the device.
2. **Settings → Security & privacy → More security settings → Encryption &
   credentials → Install a certificate → CA certificate** → **Install anyway** →
   pick the file.
3. Open Chrome and browse to `https://<EEPER_DOMAIN>/`.

**Pass:** the page loads with no certificate warning.

> Note: apps that use certificate pinning may still reject a user-installed CA;
> that is expected and does not affect the eeper PWA in the browser.

## If it fails

- Confirm the phone reaches `https://<EEPER_DOMAIN>/` (same LAN, correct
  hostname/IP in `EEPER_DOMAIN`).
- Confirm the cert you installed is the current one:
  `cd deploy && docker compose --profile core exec caddy cat /data/caddy/pki/authorities/local/root.crt`
- Regenerating the stack with a fresh `caddy-data` volume creates a **new** CA;
  re-trust it on each device.
