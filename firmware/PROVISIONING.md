# Provisioning an eeper sensor node

This walks you from a bare ESP32 to a paired sensor whose readings show up in eeper,
using the reference ESPHome configs. No prior eeper-firmware knowledge assumed. Budget
~20 minutes the first time (most of it a one-time ESPHome toolchain download).

> **What a node is.** An optional radio sensor (mmWave presence radar or a PIR) that
> publishes movement/presence readings to eeper over your WiFi. It is an _input_ to
> eeper's sleep-state insights — not a medical or alarm device.

## What you need

- An **ESP32** dev board (the configs default to `esp32dev`; set `board:` for yours).
- A sensor: an **HLK-LD2410** 24 GHz mmWave module, or any **PIR** (AM312 / HC-SR501).
- A computer with **Python 3** and [ESPHome](https://esphome.io) installed
  (`pip install esphome`), and a USB cable.
- Your eeper deployment reachable on the same WiFi, and **admin** access to its web UI.

## 1. Pair the device in eeper (get its identity)

1. In the eeper web app, sign in as an admin and open **Devices**.
2. Under **Pair a new device**, enter a name (e.g. "Crib mmWave"), pick the type, and
   press **Pair device**.
3. eeper shows the node's MQTT identity **once**: a **username** (`dev-<id>`), a
   **password**, and its **publish topic**. Copy all three now — the password is never
   shown again. The `<id>` number is the node's `device_id`.

If you lose the password, unpair the device and pair again for a fresh one.

## 2. Get the broker's CA certificate

The eeper broker is TLS-only; the node verifies it against the deployment's MQTT CA.
Copy the CA from your eeper host:

```
deploy/mosquitto/certs/mqtt-ca.crt
```

(Generated by `deploy/install.sh`. It is the same file for every node in a deployment.)

## 3. Fill in secrets and the node config

From `firmware/esphome/`:

1. `cp secrets.yaml.example secrets.yaml` and edit it:
   - `wifi_ssid` / `wifi_password` — your WiFi.
   - `eeper_mqtt_password` — the per-device password from step 1.
   - `eeper_mqtt_ca` — paste the **contents** of `mqtt-ca.crt` (keep the 2-space indent).
2. Open the config for your sensor — `eeper-mmwave.yaml` or `eeper-pir.yaml` — and set
   the substitutions at the top from step 1:
   - `device_id` — the number eeper assigned.
   - `mqtt_host` — your eeper host's address on the WiFi (IP or hostname).
   - `mqtt_user` — the `dev-<id>` username (matches `device_id`).
   - `board` (and `pir_pin` / UART pins) — to match your hardware.

`secrets.yaml` is git-ignored; never commit real credentials.

## 4. Flash the node

Connect the ESP32 over USB and run (from `firmware/esphome/`):

```
esphome run eeper-mmwave.yaml      # or eeper-pir.yaml
```

The first build downloads the esp-idf toolchain (several minutes, once). After flashing,
keep it on USB to watch the log — you want to see WiFi connect, an SNTP time sync, and
`connected to eeper broker`.

> **Why the clock matters.** Each reading is stamped with the node's unix time, and
> eeper derives a node's online/offline health from it. The firmware refuses to publish
> until SNTP has set a real clock, so if a node never appears, suspect NTP reachability
> on your network first.

## 5. Verify in eeper

Back in **Devices**, within ~15 seconds the node's health flips from **Never seen** to
**Online**, and its "Last seen" updates each heartbeat. Move in front of the sensor and
the readings update. Done — the node now feeds eeper's sleep-state insights.

## Troubleshooting

| Symptom              | Likely cause                                                                                                                           |
| -------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| Stays "Never seen"   | Wrong `mqtt_host`, bad password, or the CA doesn't match the deployment.                                                               |
| Connects then drops  | A publish outside `eeper/dev/<id>/#` — don't add custom topics or re-enable Home Assistant discovery; the per-device ACL forbids them. |
| Online but no motion | Sensor wiring / pins (`pir_pin`, UART `tx_pin`/`rx_pin`), or the radar needs a few seconds to settle.                                  |
| Time never syncs     | The node's WiFi can't reach an NTP server; allow NTP or run one on the LAN.                                                            |

## Boards ESPHome doesn't cover

If your board or sensor isn't supported by ESPHome, start from the MicroPython template
in [`micropython/`](micropython/) — it does the same job (TLS MQTT + SNTP + the contract
payload) in ~120 lines you can adapt.
